Jobs UI Redesign: Brief & Spec Proposals

Net Net V2 · prepared July 10, 2026 · companion to the interactive mockup (v10). Everything here was designed against the real netnet-web codebase and the current Product Specs; token names below are the actual variables in packages/styles/src/_variables.scss.

Contents: 1 Why · 2 Token patch · 3 Interaction grammar · 4 Screen anatomy · 5 Meters · 6 Chat indicators · 7 One-scroll rule · 8 Mobile · 9 Spec-change proposals · 10 Suggested build order

1 · Why this redesign

The audit of the live app (July 9) found one root cause behind most complaints: insufficient separation between surfaces. Cards, page background, and inner cells sit within one or two shades of each other in both themes; borders are set too faint to do their boxing work; and the decorative background pattern is painted at full strength behind content, adding noise exactly where the eye seeks structure. Second: the Tasks tab renders permanent form controls (Status, Due, Confidence dropdowns) on every deliverable row, crushing density; three deliverables fill a screen that comparable tools use for ten rows. Third: nested scrolling. The shell locks the sidebar and content area to fixed heights, so the sidebar, the content, and each task table scroll independently. Fourth: two token gaps in production, --nn-color-text-secondary and --nn-color-error, are referenced by the Jobs styles but never defined anywhere, so secondary text silently inherits the wrong color.

None of this needs a rewrite. The surfaces are one token file; the density and inline behavior are a handful of components in features/jobs/components/job-tasks/; the scroll shell is two or three files. The spec-level items are listed in section 9 with a blameless history note.

3 · Interaction grammar (applies everywhere)

  • Display text never morphs into a control. Pick lists (assignee, service type, deliverable type, status, confidence) open a styled dropdown overlay on click; dates open a calendar popover. The text stays text. This matches the spec's own display-first model (Jobs 10A.3).
  • Typed fields (task name, description, LOE) edit in place with a quiet input, no layout shift. Enter commits, Escape cancels, Tab moves on.
  • Description opens a multi-line WYSIWYG dropdown with the cursor already inside; clicking outside or Tab saves and closes; Escape cancels.
  • Inline task creation per spec 10A.4: "+ Add a task" is a text link, becomes an inline row, no Save button, autosaves when complete, Draft badge until then. New tasks start unassigned.
  • Editable cells look editable on hover (dashed underline affordance); saves confirm with a quiet "Saved ✓" flash.
  • Glyph convention: green = on, gray = off. Recurrence glyph (existing) and the new Hybrid Delivery glyph both follow it.

4 · Screen anatomy (Tasks tab)

  • Deliverable row: one line: chevron · name (+task count) · Deliverable Type pill (searchable typeahead with inline create, spec Jobs §20) · Status pill · Due (projects only) · Confidence · AI glyph (Hybrid Delivery on/off, shows aggregate AI hours when on) · chat indicator. Column-aligned so values scan vertically. The old Details wrapper is gone; Hybrid Delivery appears as a collapsible strip only when its glyph is on.
  • Task row: caret (only when multi-assignee) · status dot (click cycles) · name · description · chat · assignee · service · schedule cluster (a bounded cell holding due date and, on retainers, the recurrence glyph; recurrence on replaces the date with "Monthly") · hours bracket meter · ⋮ menu.
  • Multi-assignee: one task, N allocation rows (person + service type + LOE + actuals). Single assignee edits flat in the row; two or more expands to child rows in the same grid, all inline. "＋ Add another assignee" lives in the assignee menu; removal blocked when time is logged (spec 10A.5).
  • ⋮ menu: Edit details · Move task · Duplicate task (proposed addition) · Delete (disabled when time logged, borrowing the Quick Tasks protection).
  • Carryover: open one-off tasks from a prior month appear in the current month with an orange "↷ Jun" badge and a header counter. Prior months keep the truth (not completed then); hours logged stay in their month.

5 · The meter system

Task level, two values: the bracket meter, logged hours on the left, allocated on the right, stoplight bar between (green ≤85%, orange ≤100%, red over). Same pattern per assignee on child rows. This is V1's Hours pattern, kept deliberately.

Deliverable level, three values: the bullet meter (Stephen Few's pattern). The track length is estimated, the fill is logged (stoplight colors), and a high-contrast purple tick marks allocated. A tick short of the end shows unallocated capacity at a glance; when allocation exceeds the estimate, a red-striped extension zone continues past the track and the tick sits inside it, captioned "over-allocated by Xh."

Completion returns unused allocation: pool "allocated" counts open tasks at LOE and completed tasks at actuals. Finishing a 27h-allocated task with 18h logged returns 9h to the pool automatically, no dialog, marked on the task with a green "↩ 9h" receipt. The task's original allocation freezes as history for estimate-accuracy reporting. Reopening restores the reservation.

6 · Chat indicators (spec 9.4, locked, now rendered)

Four states, priority order: green numeric badge = unread mentions of the current user; white dot = new messages (rendered as ink in light mode to survive the background); normal contrast = chats exist, nothing new; muted = no chats. Icon always visible; counts live in the panel, not the icon. Chat exists at Job, Deliverable, and Task levels; Quick Tasks have none.

7 · One scroll per screen

The page is the only scroll surface. The job header and tabs stick; tables grow to natural height and ride the page scroll; the sidebar fits without scrolling at normal sizes. Remaining legitimate scrollers (chat panel, drawers) get thin auto-hiding scrollbars. Implementation touches: the height locks in SideNav.module.scss, .layout-children-container in globals.scss, and letting InfiniteTable render natural height (row height 40px, header 30px, down from 96px rows).

8 · Mobile

Phones get an app, not a shrunken website: a persistent bottom navigation bar with five slots, My Tasks · My Time · My Lists · Chat · Menu, where Menu opens the full-screen drawer with all sections and sub-navigation. Task rows re-compose into stacked cards; deliverable values wrap under the name; the description editor and pickers are the same overlays. Breakpoint: 768px (the existing mobileBreakpoint constant).

9 · Spec-change proposals

History note, important: several of the behaviors being corrected here (notably deliverable due dates on retainers) trace back to the original ChatGPT-drafted specifications and slipped through review. They were built faithfully to the documents. This is a process correction, not a criticism of the implementation.
P1 · Scheduling mode is exclusive and mandatory
Every task carries exactly one scheduling mode: a due date or recurrence, never both, never neither. Activating recurrence deactivates and hides the due date (the task shows "Monthly"); deactivating recurrence requires a due date. UI: the schedule cluster binds the two controls in one bounded cell. Touches Jobs 10A.2/10A.6.
P2 · Retainer deliverables have no due date
The cycle is the deadline. Remove the Due field from retainer deliverable headers, and amend Activation Review ("all Deliverables must have due dates", Jobs §8.1) with a retainer exemption. Projects keep deliverable due dates.
P3 · Cycle carryover for open one-off tasks
Never auto-complete, never silently archive. At cycle turnover, open non-recurring tasks appear in the new month with a "Carried from [month]" badge and a header counter. The task lives in one active cycle at a time; each month's actuals and completion truth stay in that month. Fills the gap Jobs §1716 explicitly defers.
P4 · Completion returns unused allocation to the pool
Pool allocated = open tasks at LOE + completed tasks at actuals. Automatic on completion, visible "↩ Xh" receipt, allocation history frozen on the task for estimate-accuracy reporting, reopening restores the reservation. New section under Jobs §11.4 (pools).
P5 · Hybrid Delivery activation moves to the AI glyph
A glyph on the deliverable row (green on, gray off, same convention as the recurrence glyph) toggles Hybrid Delivery and shows the aggregate AI hours when on. The HD editor renders as a collapsible strip that exists only while the glyph is on. The Details wrapper is removed; Deliverable Type already lives on the row as a pill. Also enforces the existing "collapsed by default" rule (Jobs §2474) the current UI violates.
P6 · Enumerate the task ⋮ menu
The Jobs spec names the menu but never defines it. Proposed contents: Edit details · Move task · Duplicate task (new) · Delete, with Delete disabled while any assignee has time logged (adopting the Quick Tasks rule).
P7 · General Job Tasks group is removed
Per the July 10 decision: the special group goes away unless client demand warrants it. Owners who want job-wide work create an ordinary deliverable (e.g., "General"), which then has a real pool, closing the unpooled-effort hole. Supersedes Jobs §10.6.
P8 · Deliverable meters adopt the bullet pattern
Track = estimated, fill = logged, purple tick = allocated, striped extension = over-allocation, as in section 5. Task and assignee meters use the bracket pattern. Meter thresholds unchanged (85/100).
P9 · Interaction grammar codified
Section 3 (no-morph overlays, in-place typed edits, description editor behavior, autosave/Draft, hover affordances, keyboard rules) becomes normative language in Jobs 10A, replacing the current always-rendered controls on deliverable headers.
P10 · Token additions and surface roles
Add the two missing tokens (--nn-color-text-secondary, --nn-color-error) in both themes; adopt the section 2 values; demote the background pattern; document the three-layer surface system (bg → card → soft → light) in Net-Net-V2-Resources.

10 · Suggested build order

  1. Token patch + missing tokens + pattern demotion (one file, whole-app payoff, easiest QA).
  2. One-scroll shell (sidebar height, content container, table natural height).
  3. Deliverable row compaction (pills replace permanent controls; AI glyph; HD strip; Type pill).
  4. Task grid (bracket meters, schedule cluster, inline grammar, child allocation rows, ⋮ menu).
  5. Pool bullets + allocation return + carryover (needs backend: P3, P4).
  6. Mobile bottom nav + card layouts.
